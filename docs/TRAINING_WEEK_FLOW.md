# Training Week Flow – Auto-Generate & Store

Training week is **no longer hit manually** from the frontend. It is generated by the backend and stored in the DB.

---

## 1. When is it generated?

| When | What happens |
|------|----------------|
| **Right after onboarding** | When the user completes onboarding (PUT /api/profile with full data including **availability**: `training_days_per_week`, **preferred_rest_days**), the backend triggers **first week** generation in the background. No manual API call. |
| **Every Sunday (cron)** | At **00:00 Sunday** (server time), a cron job runs and pre-generates **next week** for every user who has completed onboarding. |

**Change of days:** If the user later changes e.g. from 5 to 3 training days in profile, that change applies from the **next week**. The current week stays as-is. The API can return: *"Your next training week will have 3 sessions."*

---

## 2. How is the week stored?

One document per user per week: **WeeklyTraining**.

- **week_start:** Sunday 00:00 of that week.
- **days:** Object with keys **monday, tuesday, wednesday, thursday, friday, saturday, sunday**.
  - Each day is either:
    - **`{ type: 'rest' }`** – rest day (from onboarding **preferred_rest_days**).
    - **`{ type: 'training', exercises, coach_note, key_cues, ... }`** – training day (from AI + onboarding).

Example:

```json
{
  "week_start": "2025-02-09T00:00:00.000Z",
  "days": {
    "monday": { "type": "training", "exercises": [...], "coach_note": "...", "key_cues": [...] },
    "tuesday": { "type": "training", "exercises": [...], ... },
    "wednesday": { "type": "rest" },
    "thursday": { "type": "training", ... },
    "friday": { "type": "training", ... },
    "saturday": { "type": "rest" },
    "sunday": { "type": "rest" }
  },
  "profile_snapshot": { "training_days_per_week": 4, "preferred_rest_days": ["Wednesday", "Saturday", "Sunday"], "session_duration": 60 }
}
```

Data for each day comes from:
- **Onboarding:** training days per week, **preferred_rest_days**, session duration (all from DB).
- **AI:** exercises, coach_note, key_cues, etc., using the Training Logic doc + profile.

---

## 3. What the frontend does

- **Do not** call POST /api/training/generate from the app for the main “my week” view.
- **Do** call **GET /api/training/week** (with auth). Response is the **current** (or latest) stored week:
  - **week_start**
  - **days** (monday..sunday, each with `type: 'training' | 'rest'` and, for training, exercises, coach_note, key_cues, etc.)
  - **is_first_week**, **profile_snapshot** (optional).

If no week exists yet (e.g. onboarding just finished and generation is still running), the API returns `data: null` and a message; the app can retry or show “Your week is being generated”.

---

## 4. Summary

| Item | Detail |
|------|--------|
| **First time** | Generated automatically after onboarding (PUT /api/profile with availability). |
| **Later weeks** | Generated every **Sunday** by cron; stored for the **next** week. |
| **Payload / data** | All from DB: onboarding (days, rest days, session duration) + AI (exercises, notes, cues). |
| **Frontend** | Use **GET /api/training/week** only; no manual generate for the main flow. |
| **Changing days** | Takes effect from the **next** week; current week is unchanged. |
